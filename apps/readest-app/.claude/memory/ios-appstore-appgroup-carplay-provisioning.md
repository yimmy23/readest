## ACTUAL ROOT CAUSE of the ORIGINAL #4891/#5188 break (confirmed) — stale skip-worktree pbxproj
`src-tauri/gen/apple/Readest.xcodeproj/project.pbxproj` is marked **skip-worktree** (`git ls-files -v` → `S`) and its committed/HEAD blob still has `CODE_SIGN_ALLOW_ENTITLEMENTS_MODIFICATION = YES` on BOTH extension targets (count 2, lines ~412/626); the committed pbxproj is also severely STALE — predates the ReadestWidget target entirely (no widget sources, no appex embed, no CarPlay). #4891 (Jul 2) removed the flag from `project.yml` ONLY. Tauri's iOS build does NOT re-run xcodegen (`load_pbxproj`+save mutates signing only), and skip-worktree means the locally xcodegen-regenerated (flag-free) pbxproj NEVER gets committed → git/CI keep the flagged, stale one. At 0.11.18's build (Jul 7) nobody had run `xcodegen generate` since #4891, so the on-disk pbxproj still had the flag → App Store archive/export re-sign stripped `group.com.bilingify.readest` from both `.appex` → widget placeholder / share only default group. The flag's strip fires during app-store archive/export (automatic re-sign minimizes extension entitlements), NOT dev builds — why `dev-ios` works. NOTE the tracked project is `Readest.xcodeproj` (scheme `Readest_iOS`), NOT `Readest_iOS.xcodeproj` — grepping the wrong name returns 0 = file-not-found, not "flag absent".

Currently SELF-HEALED on this machine: CarPlay #5085 required a `xcodegen generate`, regenerating the pbxproj from the flag-free project.yml → on-disk pbxproj now has 0 flag occurrences + full widget/CarPlay targets → app-store re-sign of the current archive shows App Group PRESENT in both extensions → **next release fixes the widget, no portal change.**

This blocking guard is the real backstop.

**DEAD END — do NOT run `xcodegen generate` in the release script.** xcodegen expands `${VAR}` from the GENERATING shell's env; `pnpm`/CI set `FORCE_COLOR` (=3), so it baked a literal `3` into the "Build Rust Code" script phase (`--configuration release 3 ${ARCHS}`), breaking the build ("Arch specified by Xcode was invalid. {arch} isn't a known arch", xcodebuild exit 65) — broke `dev-ios`. If you MUST regen, run `env -u FORCE_COLOR xcodegen generate` in a clean shell and commit. Restore a broken pbxproj by re-copying the known-good one (project.yml's script phase must keep literal `${FORCE_COLOR}`). Also skip-worktree: `ShareExtension/Info.plist`. If keeping the pbxproj commit, re-set `git update-index --skip-worktree` after to avoid per-build signing churn (guard still protects releases; the flag can't return via the now-clean project.yml).

## External parallel: Reflect PR #612 (same symptom, DIFFERENT root cause)
`team-reflect/reflect-open` #612 (merged 2026-07-06) fixes the identical symptom class — an iOS app-extension loses its App Group entitlement ONLY in CI TestFlight/App-Store builds (works in dev), so at runtime the appex reads an empty App Group (their ShareExtension → `containerUnavailable`, "Couldn't save"). But their diagnosed cause is DIFFERENT from ours: Tauri's App-Store-Connect API-key flow archives with `CODE_SIGNING_ALLOWED=NO` and `tauri-cli` dummy-signs ONLY the app binary, so the appex enters `-exportArchive` UNSIGNED → export re-signs it with an auto-derived minimal set (no App Group). Their fix = a ShareExtension post-build phase that AD-HOC SIGNS the appex WITH its entitlements when `CODE_SIGNING_ALLOWED=NO` (readest has NO such phase — only `codeSign: true` on embed). Ours = REMOVE `CODE_SIGN_ALLOW_ENTITLEMENTS_MODIFICATION` (entitlement minimization, not unsigned-appex). Our re-export verification showed the group PRESENT after only removing the flag → our appex was NOT entering export unsigned, so we hit a different Tauri path; #612 does NOT invalidate #5188. Both teams independently added the SAME backstop (readest `verify-ios-appstore-entitlements.sh` iterates BOTH `ReadestWidget`+`ShareExtension` via `codesign -d --entitlements`, `set -euo pipefail`; Reflect `assertIpaAppexEntitlements`) — strong signal it's the right guard. #612's pre-sign-the-appex phase is the RECURRENCE PLAYBOOK: if the widget/share breaks again with the flag already absent (e.g. after a Tauri bump onto the dummy-sign path), pre-sign the appex, don't hunt the flag. Reflect names it an upstream `tauri-cli` bug (dummy-signs only the app binary, never embedded appexes) — worth co-filing.

## Verification recipe (fast, no full rebuild, no upload)
Re-export the existing archive with an app-store ExportOptions — reproduces the real release re-sign (build.rs uses `signingStyle: automatic` + `.allow_provisioning_updates()` for archive L448 AND export L556; method `app-store-connect`):
```
xcodebuild -exportArchive -archivePath src-tauri/gen/apple/build/Readest_iOS.xcarchive \
  -exportPath OUT -exportOptionsPlist <method=app-store-connect,signingStyle=automatic,teamID=J5W48D69VR> \
  -allowProvisioningUpdates -authenticationKeyID $APPLE_API_KEY -authenticationKeyIssuerID $APPLE_API_ISSUER -authenticationKeyPath $APPLE_API_KEY_PATH
codesign -d --entitlements :- OUT/Payload/Readest.app/PlugIns/<ext>.appex | grep application-groups
```
Result get-task-allow=0 + beta-reports-active=1 = real distribution signing. ASC API: ES256 JWT from `.env.ios-appstore.local` (`APPLE_API_KEY`/`APPLE_API_ISSUER`/`APPLE_API_KEY_PATH`); `/v1/apps?filter[bundleId]=` for live build list (live App Store = 0.11.18, Jul 7, post-#4891 but still broken).
