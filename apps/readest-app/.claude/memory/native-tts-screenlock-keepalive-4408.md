# Android system TTS screen-lock stall (#4408, second half)

**Symptom (reporter, GrapheneOS/Vanadium):** offline system TTS (Supertonic/Sherpa/GOS) reads ~a page then STOPS with the **screen locked**, app "frozen for minutes"; screen ON = fine; **Edge TTS reads fine screen-locked**. Distinct from the [[native-tts-offline-autoadvance-4613]] error-skip fix (PR #4716) which was the screen-ON dead-end.

**Root cause:** system/native TTS renders audio in the EXTERNAL engine process, NOT the WebView. Its per-sentence auto-advance runs in WebView JS (`NativeTTSClient.speakMark`: per-sentence `invoke('speak')` + `setTimeout(100)` poll + plugin `tts_events`). Screen lock -> page `hidden` -> Chromium throttles timers (measured ~1s coalescing on Pixel/API16 emu; ~60s + frozen fetch on stricter devices, [[cdp-android-webview-profiling]]) and FREEZES task queues on stricter builds -> the JS loop stalls after the queued sentence. **Edge is exempt because its audible WebAudio output makes the page "audible"** (WebAudioPlayer.ts already noted onended is "background-safe"); audibility is a per-page property Chromium derives from actual non-silent output. The MediaPlaybackService foreground service + silent ExoPlayer keep-alive keep the PROCESS/audio-focus alive but do NOT make the WebView page audible.

**Fix (MERGED-pending, local commit `6888e66dd` on dev):** play a continuous **inaudible** WebAudio tone (40 Hz, gain 0.0008 ~ -62 dBFS) on the shared AudioContext while a direct-speak engine reads on Android, so the page stays "audible" and its timers keep running screen-off — the same exemption Edge gets for free. `WebAudioPlayer.ts`: `startAudioKeepAlive`/`stopAudioKeepAlive` (module-singleton osc->gain->destination; guarded `typeof AudioContext==='undefined'` for jsdom). `TTSController`: `#syncNativeAudioKeepAlive()` gated `appService.isAndroidApp && ttsClient.getCapabilities().mediaClock===false`; START after `state='playing'` in `#speak` (idempotent across per-paragraph transit stops — do NOT hook raw `stop()`); STOP in `pause()`, `#terminate()`, `shutdown()`. iOS EXCLUDED on purpose (native audio session owns playout; a WebKit now-playing client would fight it, see [[native-ios-tts-4676]]). Edge = no-op (mediaClock true). Tests in `tts-controller.test.ts` "native TTS background keep-alive (#4408)" mock `@/services/tts/WebAudioPlayer` via `importActual` spread; verified fail-without/pass-with by neutralizing the `#speak` hook.

**Why non-silent, not the existing silent keep-alive:** `unblockAudio()` (silent `<audio>`, [[ttsMediaBridge]]) and a resumed-but-idle AudioContext do NOT earn the exemption — Chromium keys off measured non-silent OUTPUT. Emulator threshold: gain 0.0001 already exempted; 0 (true silence, connected osc) exempted in 45s but may be detected silent long-term, so ship non-zero. 40 Hz = below phone-speaker reproduction + masked by speech.

**Emulator verification recipe (Pixel_9_Pro AVD, Android16, arm64):** device does NOT reproduce the severe freeze (only mild ~1s throttle in 60s, like the Xiaomi non-repro) but DOES show the throttle + that the fix removes it. (1) start real FGS via CDP `invoke('plugin:native-tts|set_media_session_active',{payload:{active:true,keepAppInForeground:true,...}})` (installed 0.11.17 needed `keepAppInForeground`, since removed). (2) Throttle probe: `setInterval(200)` recorder -> `adb shell input keyevent 26` (lock) 60s -> keyevent 224 + `wm dismiss-keyguard` -> read `window.__ticks` gaps (CDP evaluate HANGS while locked; read AFTER unlock). Test A (FGS, no WebAudio) = 1s coalescing; Test B/C (audible osc, even 0.0001) = stays 200ms. (3) End-to-end: install fresh build, open book via `am start VIEW content://media/external/file/<id> -t application/epub+zip` (needs bundled frontend; the dev-server debug APK fails on 192.168.x:3000), force system TTS via `localStorage ttsPreferredVoices={"preferredClient":"native-tts"}`, start via deep link `am start VIEW readest://book/<hash>?autoplay=tts` (book already open -> starts TTS). Google TTS synthesizes OFFLINE on the emu (`en-us-x-iog-seanet-embedded`). Result: 22 synth requests + 6 `[TTS] speak` continuous across a 75s lock; recorder 476 ticks mostly 200ms (vs Test A 1s). NOTE: online-vs-offline is irrelevant to the bug — audio is external either way.

**CDP helper:** node ESM importing `ws` from monorepo needs absolute path + `const {WebSocket}=pkg` (CJS); pass `userGesture:true` to `Runtime.evaluate` for `AudioContext.resume()`. macOS `date +%N` unsupported (use `adb shell log -t TAG` markers to bound the logcat window).
