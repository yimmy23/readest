# iOS TTS media session: native MPNowPlayingInfoCenter (2026-07-12)

## PAUSE-DROP ROOT CAUSE (device log 10:21:18): KVC playbackState HARMFUL on device
With native playout, device test passed EVERYTHING except pause (card dismissed; AirPod play resumed QQMusic). Pause-moment log: client was correctly INFERRED Paused (PlaybackState Playing→Paused — the architecture works!), BUT `CanBeNowPlayingPlayer changed to false` flapped at the exact pause instant → `ActiveNowPlayingClient changed ... to client-(null)` — slot vacated, never re-granted (Paused clients don't re-elect). Culprit: the KVC `playbackState=2` write in the pause push — INERT for forcing Playing on device but NOT inert for paused/stopped: it resigns player eligibility. FIX: `setSystemPlaybackStateRaw` is now `#if targetEnvironment(simulator)` only (sim still NEEDS it as the decisive Playing lever). Also: keep-alive now runs CONTINUOUSLY through pauses (session I/O keeps the client remembered; declared rate-0 freezes the card timeline); it stops only on teardown.

## ENDGAME (2026-07-13): NATIVE PCM PLAYOUT — implemented, device verify pending
User-approved after pause-hold kept failing ("commit to the native PCM playout plan"). Edge TTS MP3 utterances now play through an in-process AVPlayer on iOS instead of WebAudio — the audio finally lives in the session that owns Now Playing, so card/pause-slot/AirPods/mute-switch are all textbook music-app behavior. ARCHITECTURE: Swift dumb player in NativeTTSPlugin (`playout_control` start-session/end-session/abort/pause/resume/set-rate + `playout_enqueue` base64-MP3→tmp file→AVPlayerItem, `.timeDomain` pitch algorithm for pitch-preserving rate, returns durationMs + `playout_position`; `playout_events` chunk-start/session-end; inter-sentence gap on a NATIVE Timer so bg JS throttling can't stall it; single AVPlayer, manual advance). JS `NativeAudioPlayer` (src/services/tts/NativeAudioPlayer.ts) mirrors WebAudioPlayer's session interface (startSession/scheduleRawChunk(index!)/waitUntilReady backpressure MAX_PENDING=3/endSession/abortSession/pauseContext/resumeContext/getPlaybackPosition SYNC via 250ms poll + rate-scaled extrapolation — AVPlayer.currentTime IS media time so Edge boundaries need no rescaling; trimStartSec=0, no decode/trim/WSOLA on this path). EdgeTTSClient selects it on iOS Tauri; setRate applies LIVE natively. claimAudioSession back to NON-mixable []; unblockAudio iOS = plain early return (no element, no audioSession.type). Tests: native-audio-player.test.ts (7), edge client tests pin web path via env mocks. VERIFY ON DEVICE: card while playing, PAUSE holds card, AirPod resume, mute switch, word highlight sync, rate change, background long-play, sentence gaps.

The 0.11.18 iOS regression "lock screen media controls dead during TTS": the WebAudio engine ([[edge-tts-webaudio-engine]] #4931) removed the Edge `<audio>` element (chunks connect straight to `ctx.destination`), and **without an HTMLMediaElement WebKit publishes NOTHING to the system now-playing** - `navigator.mediaSession` metadata/handlers are inert. Lock screen AND CarPlay read the same `MPNowPlayingInfoCenter`, so both were dark.

## Dead end (tried, reverted): MediaStreamAudioDestinationNode -> <audio> element
Routing the shared WebAudio graph through `createMediaStreamDestination()` + an `<audio srcObject>` element DID restore now-playing (metadata + transport), BUT:
- The element's own stream clock (unrelated to book position) fought `setPositionState` - lock-screen timeline jumped/messed, CarPlay timeline ran while paused (two-writer problem; WebKit publishes element time).
- A suspended AudioContext starves the live stream -> the element renders underruns as a repeating ~50ms glitch from the speaker while paused.
Any audible element makes WebKit a second now-playing writer; unfixable while the element exists.

## Final architecture: native session, single writer
`getMediaSession()` (`src/libs/mediaSession.ts`): iOS Tauri -> `TauriMediaSession` (same as Android). The native-tts Swift plugin's media methods (dormant since the #4676 revert) do everything: `set_media_session_active` -> MPRemoteCommandCenter play/pause/toggle/next/prev; `update_media_session_metadata` -> title/artist/album + base64 artwork (data URL decoded to MPMediaItemArtwork); `update_media_session_state` -> rate 0/1 + elapsed/duration (ms->s). WebAudio stays direct-to-destination; no element. **The #4676 "iOS must NOT route native" verdict is obsolete** - it predated the element removal (back then Edge's <audio> made WebKit the richer owner). VERIFIED in sim (mediaremoted log): `client-com.bilingify.readest ... setting nowPlayingItem`, sentence title, CalculatedPlaybackPosition = section timeline advancing, rate 1.0; `CarPlay: Resolved now playing identifier to com.bilingify.readest`. An app CAN be Now Playing with WKWebView WebAudio + active .playback AVAudioSession + nowPlayingInfo + remote targets.

## Lock-screen card: the THREE required levers (found empirically, sim iOS 18.5)
The native path needs ALL of these in the Swift plugin (`NativeTTSPlugin.swift`) or the lock screen shows NO card at all:
1. **Silent in-process keep-alive player** (`AVQueuePlayer` + `AVPlayerLooper` looping a generated 1s silent WAV in tmp; play/paused in lockstep with TTS via `update_media_session_state`) - the app session must carry real audio.
2. **`UIApplication.shared.beginReceivingRemoteControlEvents()`** on activate (end on deactivate).
3. **KVC `playbackState` on MPNowPlayingInfoCenter.default()** - THE decisive lever: `center.setValue(NSNumber(1|2|3), forKey: "playbackState")` (playing/paused/stopped), guarded by `responds(to: "setPlaybackState:")`. The property is macOS-only in headers but exists in the iOS runtime. NOTHING else flips the MediaRemote client to Playing (proven: playing AVAudioPlayer + active .playback session + rate-1 info + active MPNowPlayingSession all leave it Paused, and a Paused client gets NO lock card). App Store gray zone - documented, trivially removable via the guard.

**MPNowPlayingSession (iOS 16+) tried and REVERTED**: it registers a SECOND MediaRemote player for the client; item landed on the session player while state/commands sat on MediaRemote-DefaultPlayer -> card without transport buttons. ONE surface only: `.default()` info center + `.shared()` command center + KVC state -> everything on DefaultPlayer; log confirms supportedCommands all enabled=1 + Playing + ElectedPlayer=Readest.

**Lock-card button icons**: the card renders ICONS only for SKIP-INTERVAL commands (`skipForward/skipBackwardCommand`, `preferredIntervals=[10]` -> the ±10 glyphs) - prev/nextTrack commands alone leave the button row blank-but-tappable. So the plugin registers skip commands routed to the sentence-level seek handlers (`media-session-seek-forward/-backward` events -> JS `seekforward/seekbackward` -> `controller.forward(true)/backward(true)`), matching Apple Books' audiobook UX. **The center play/pause glyph never renders in the SIMULATOR under ANY architecture** (WebKit era had the same gap; hit target works, state is published correctly) - sim rendering gap only; renders on real devices.
