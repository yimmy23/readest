Tauri 2.11 (fork bump readest-2.11, upstream #15266 security fix) enforces ACL for REMOTE origins even without an AppManifest: custom (non-plugin) `generate_handler!` commands, which previously bypassed ACL entirely, are denied from remote origins with `"<cmd> not allowed. Plugin not found"`. The webdriver test harness trips it because the vitest browser tester page is served from vitest's own port (http://127.0.0.1:63315) which tauri classifies as remote (`is_local_url` = tauri:// protocol, devUrl/frontendDist-relative, or app-registered custom scheme). Broke `pnpm test:tauri` (56 failures: epub-parser-parity, tauri-app-service, smoke — all app-command invokes, e.g. get_executable_dir at NativeAppService init) deterministically on CI build_tauri_app and locally. PRODUCTION UNAFFECTED: app pages are local origins where no-manifest bypass still applies.

**Fix (PR #5085):** three pieces, ALL required (verified by matrix: no-manifest = fail; manifest + local grants only = fail; + remote capability = 112 pass):
1. `src-tauri/build.rs`: `tauri_build::try_build(Attributes::new().app_manifest(AppManifest::new().commands(&[...24 commands...])))` — generates `allow-<kebab-case-cmd>` permissions into `src-tauri/permissions/autogenerated/` (committed). KEEP IN SYNC with generate_handler! in lib.rs.
2. `capabilities/default.json`: grants all app allow-* to local windows (REQUIRED: once has_app_acl_manifest, LOCAL windows also resolve app commands via ACL — missing grants would break the real app).
3. `capabilities/webdriver-remote.json`: `"remote":{"urls":["http://localhost:*","http://127.0.0.1:*"]}`, local:false, windows [main], same allow-* list. Gated: `tauri.conf.json` pins `app.security.capabilities: ["default","desktop-capability"]` (webdriver-remote inert in shipped builds); `scripts/test-tauri.sh` selects it via `--config '{"app":{"security":{"capabilities":[...,"webdriver-remote"]}}}'`.

Gotchas: `app.security.capabilities` pinning means NEW capability files must be added to that list or they silently don't apply (safe direction). Capability selection is baked at compile time — changing --config capabilities triggers a rebuild. Permission ids are kebab-case of snake_case command (`allow_paths_in_scopes` → `allow-allow-paths-in-scopes`). Plugin commands passing from the same remote origin in some probe runs remains unexplained curiosity (dev-build resolve nuance); does not affect the fix. Debugging trick: instrument fork `on_message` with eprintln of (request.cmd, request.url, is_local).
