# CarPlay TTS support (iOS)

**DEVICE CRASH + FIX (2026-07-12 evening):** on a REAL iPhone (iOS 18.7.9) the app SIGSEGVed ~1.3s after launch — crash report: `objc_retain` inside `-[UIApplication _connectUISceneFromFBSScene:transitionContext:]`. ROOT CAUSE = **use-after-free in upstream tao 0.35.3**: `configuration_for_connecting_scene_session` (src/platform_impl/ios/view.rs ~L646) returned `Retained::as_ptr(&config)` — a raw borrow of a `UISceneConfiguration` dropped (released) when the fn returned; UIKit then retained the freed object. Only triggers when the app has a `UIApplicationSceneManifest` (our CarPlay scene) since ordinary Tauri apps never route through that callback; the SIMULATOR survived by allocator luck. FIX: `packages/tao` is a SUBMODULE of github.com/readest/tao branch `readest-0.35` (= tag tao-v0.35.3 + fix commit da30a3b9; was vendored crates.io copy first, swapped on PR #5085) + root `Cargo.toml [patch.crates-io] tao = { path = "packages/tao" }` (same pattern as tauri submodule); the return became `Retained::autorelease_return(config)` (objc2 0.6 +0 convention via objc_autoreleaseReturnValue). NOTE: the repo manifest path-deps `tao-macros` (subdir), so Cargo.lock gains a path `tao-macros 0.1.3` entry that the crates.io tarball did not have. **CAP-LINTS GOTCHA:** cargo passes `--cap-lints allow` to registry/git deps but NOT to path deps, so a path-patched tao builds with warnings live; Linux CI (`-D warnings`) failed on glib's deprecated `MainContext::channel` (3 sites: linux/event_loop.rs x2, linux/portal.rs). RESOLVED workflow-side (user wanted fork parity): the deny came from actions-rust-lang/setup-rust-toolchain's DEFAULT `rustflags: -D warnings` (only injected when RUSTFLAGS unset; rust_lint job presets RUSTFLAGS so was immune); build_tauri_app now sets `rustflags: ''` in pull-request.yml. release/nightly use dtolnay/rust-toolchain = no injected deny. Fork branch readest-0.35 force-reset to da30a3b9 = tag + UAF fix ONLY. Any warning in ANY path-patched dep can break -D warnings jobs; a git-dep [patch] would be lint-capped instead. **Worth an upstream PR to tauri-apps/tao.** Other `Retained::as_ptr` uses in tao ios are safe (borrow-during-msg_send; WindowId is an opaque identity).

STATUS UPDATE (2026-07-12, supersedes BLOCKED below): **WORKING (sim; device pending the tao UAF fix verification).** User upgraded the tauri fork deps (tao 0.34.8 -> 0.35.3, wry 0.54 -> 0.55, tauri 2.11.5 + @tauri-apps/api 2.11.1 - which surfaced a latent `unlisten_js_script` unguarded `listeners[eventId].handlerId` TypeError, fixed with a guard in `packages/tauri/crates/tauri/src/event/mod.rs`). With tao 0.35's UIWindowScene support (tao#1154), the scene manifest was RE-ADDED (CarPlay-only role, SupportsMultipleScenes=true) and the phone renders fine - **the predicted tao `configuration_for_connecting_scene_session` role patch was NOT needed** (iOS falls back to the plist config for the CarPlay role; tao's dynamic config only claims the window role in practice). Verified in sim: phone renders, Readest on CarPlay home, CPListTemplate row (title/author/cover), CPNowPlayingTemplate with working transport + timeline. Media state/artwork/transport now flow via the NATIVE media session - see [[ios-tts-media-session-native]] (WebKit navigator.mediaSession was a dead end after the WebAudio engine removed the <audio> element). Remaining polish: skipForward/skipBackward + changePlaybackPosition commands in Swift.

ORIGINAL BLOCKER ANALYSIS (historical, pre-upgrade): **BLOCKED on wry/tauri UIScene adoption.** App-side code is complete and CarPlay itself works (Readest appears on the CarPlay home screen, verified in the iOS 18.5 Simulator via Xcode's I/O > External Displays > CarPlay), BUT the required `UIApplicationSceneManifest` flips the whole app into the UIScene lifecycle and **wry creates the iOS UIWindow via the legacy app-delegate path and never attaches it to the auto-created UIWindowScene -> the phone has no application key window and renders BLACK for every user** (simulator log: `<UIWindowScene>` present + WebKit processes launched but `currentApplicationKeyWindow: 0x0`; NativeBridge spams "No key window found"; WebViewLifecycleManager loops about:blank -> reload -> `-999` cancelled). `SupportsMultipleScenes` true/false makes NO difference - the manifest's mere presence causes it. The research's "CarPlay-only scene keeps the app-delegate window" recipe does NOT hold for wry (wry is strictly non-scene).

DECISION (2026-07-12): removed the scene manifest from `dev` (commit `e6c7b6d96`, edits the tracked `src-tauri/Info-ios.plist` seed) so the iOS app is not broken. The rest of the CarPlay code stays DORMANT on `dev` (base was `c71bf3f0d`): `update_carplay_state` command, JS bridge, `CarPlaySceneDelegate.swift`, `carplay-audio` entitlement, CarPlay/MediaPlayer framework links. **Re-enabling CarPlay is gated on making the wry iOS window scene-aware** - either an in-app `UIWindowSceneDelegate` that adopts wry's window into the scene (`window.windowScene = scene; makeKeyAndVisible()`), or a patch to the `readest/tauri` fork (`packages/tauri`) iOS window creation. Same UIScene adoption iOS 26 will force anyway. Next step per user: investigate the UIWindowSceneDelegate-adopts-wry-window approach.

## Scope (deliberately minimal, mirrors Android Auto)
Only surfaces the CURRENTLY-reading book (gated on active session) + a placeholder when idle. `CPListTemplate` root -> tap -> `CPNowPlayingTemplate.shared`. NO library browsing / start-a-book-from-car (same limitation Android Auto has). App Store review of a browse-less audio app is a flagged risk.

## Architecture (how it hangs together)
- Now Playing screen + ALL transport (play/pause/skip/seek) come FREE from `MPNowPlayingInfoCenter`/`MPRemoteCommandCenter`, already driven by the WKWebView `navigator.mediaSession` during TTS. No CarPlay button is routed by hand.
- New iOS-only Tauri command `plugin:native-tts|update_carplay_state({active,title,author})` (native-tts plugin: models/commands/mobile/desktop-stub/lib/build.rs COMMANDS/permissions). JS `notifyCarPlayState()` (`src/services/tts/carPlaySession.ts`, gated `getOSPlatform()==='ios' && isTauriAppPlatform()`) is fired from `TTSMediaBridge.bind()` (active) / `unbind()` (inactive, INSIDE the `if(this.#controller)` guard so a fresh bind's internal unbind() does not emit spurious inactive).
- Swift handler writes `UserDefaults.standard` keys `readest.carplay.{active,title,author}` + posts `NotificationCenter` name `readestCarPlayStateChanged`. It DELIBERATELY does NOT touch MPRemoteCommandCenter / audio session / MPNowPlayingInfoCenter (keeps the iOS lock screen owned by navigator.mediaSession - avoids the #4676 regression class [[native-ios-tts-4676]]).
- `CarPlaySceneDelegate.swift` lives in the MAIN app target (`src-tauri/gen/apple/Readest_iOS/`), plain `final class ... : UIResponder, CPTemplateApplicationSceneDelegate` (no custom `@objc(...)` name) resolved from the plist as `$(PRODUCT_MODULE_NAME).CarPlaySceneDelegate` (PRODUCT_MODULE_NAME=Readest). Reads the UserDefaults keys + `MPNowPlayingInfoCenter` artwork; rebuilds its list on the notification.

## GOTCHAS (non-obvious, cost real time)
- **iOS Info.plist edits go in the TRACKED seed `src-tauri/Info-ios.plist`** (referenced by `tauri.conf.json` `iOS.infoPlist`), NOT the generated `src-tauri/gen/apple/Readest_iOS/Info.plist` which is gitignored + regenerated (Tauri merges the seed in; edits to the generated file get wiped). The whole `src-tauri/gen` tree is gitignored but a specific set is FORCE-tracked (project.yml, Readest_iOS.entitlements, extension Swift sources, pbxproj); the generated main-app Info.plist is intentionally NOT tracked. New app-target Swift (CarPlaySceneDelegate.swift) is force-added like the ShareExtension sources.
- Scene manifest is CarPlay-ONLY (`CPTemplateApplicationSceneSessionRoleApplication`, `UISceneClassName=CPTemplateApplicationScene`; NO `UIWindowSceneSessionRoleApplication`) so wry's app-delegate window keeps the phone UI. `UIApplicationSupportsMultipleScenes=true` (the one uncertain knob - flip to false if the phone screen goes black in the simulator). iOS 26 will eventually force UIScene adoption on the phone window (Tauri/wry's change, not ours).
- entitlement key `com.apple.developer.carplay-audio`; project.yml links `CarPlay.framework` + `MediaPlayer.framework` and sets `SWIFT_VERSION 5.0` (first Swift in the Obj-C++ main target).
- `CPNowPlayingTemplate` CANNOT be a root template; root must be `CPListTemplate`. No public notification for `MPNowPlayingInfoCenter` changes (hence the JS bridge).
- Rebase gotcha (hit during rebase onto origin/main): worktree foliate-js submodule uses a local `file://` mirror that git blocks by default -> `git -c protocol.file.allow=always submodule update packages/foliate-js`. See [[worktree-rebase-submodule-drift]].
